# 開発日誌 2026-07-22

オンラインたいせんモード(PeerJS/WebRTC)の Phase 2〜3 を実装。

---

## 決定事項

- ホスト権威型モデルを踏襲: ホストが試合を100%シミュレーションし、ゲストは自分の入力を送るだけ・状態を受信して描画するだけ(設計書 soccer-game-online-mode-prompt.md どおり)
- 赤チームに`controlledDefender`(青の`controlledPlayer`の赤版)を新設。ボールを持った赤選手に自動で切り替わる仕組みを青と対称に実装し、オンライン時はゲストの入力をそこに流し込む方式にした
- 通信プロトコルは設計書の簡易版(me/you)ではなく、選手・DF・青GK・スコア・タイマーをまるごと20Hzで送る実装のまま踏襲(Phase 1〜2で既にその方針だったため)
- ゲスト側はポーズ/せってい操作に触れない(タップは常にフィールド座標に変換してホストへ転送するだけ)と決定。ゲストがローカルで誤ってポーズ状態になる副作用も同時に解消

## 実施したこと

- Phase 2: 接続確立後、ホストの「しあいの せってい へ」ボタンから通常のたいせん設定フロー(時間/なんいど選択)に入れるように。ホストは試合状態を20Hzでゲストへ送信、ゲストは受信してそのまま描画(結果画面もゲストは観戦のみ)
- Phase 3: ゲストのスティック/キー/傾き入力を30Hzでホストへ送信 → ホストが`controlledDefender`をその入力で動かす(他の赤選手は従来どおりAI)。ゲストのタップをフィールド座標に変換して送信 → ホストが赤のシュート/パスとして判定(青のタップ処理と対称ロジック)
- 検証: ブラウザ環境を丸ごとスタブする代わりに、追加した関数(入力反映・タップ判定・操作対象切り替え)をコードから直接抽出し、モックしたオブジェクトに対して7項目のユニットテストを実施 → 全合格
- コミット: 6be1826「オンラインたいせんモード Phase 2〜3: 状態同期とゲスト操作を実装」

### 追記: オンラインしあいモード(8on8)対応

- 方針転換の決定: ユーザーの希望により、オンライン対戦の本命を「たいせんモード(3on3相当)」から「しあいモード(8on8)」へ変更。3on3のオンラインは動作確認済みだが今後の必須ではない。設計書v1では対象外としていたしあいモードのオンライン化を実施
- たいせんモードで確立したホスト権威型の仕組みをそのまま流用:
  - オンライン接続後、ホストの「しあいの せってい へ」→ s7Setup → s7Startで開始(従来のたいせんモード遷移を変更)
  - ホストが全16人をシミュレーション。赤のボール保持者(controlledRed)だけAIの代わりにゲストの入力で動かし、残りの赤はAIのまま
  - 全選手位置+ボール+スコア+タイマー+保持者/操作対象インデックスを20Hzで送信(t:'s7st')。ゲストは受信描画+カメラのみローカルでボール追従
  - ゲストの入力(30Hz)・タップを送信。タップはフィールド/ミニマップ座標に変換し、ホストが赤のシュート(下ゴール)・パスとして判定
- ゲスト視点対応を追加: 操作中の赤選手をむらさき強調、ミニマップ強調・パス合図を赤基準、スコア/勝敗を「あなた=赤」に入れ替え表示、青のパスコースはゲストに非表示
- 試合終了時に結果画面遷移を強制ブロードキャスト(通常送信は間引かれるため取りこぼし防止)。たいせんモードにも同じ修正を反映
- 検証: s7オンラインの追加関数を7項目ユニットテスト → 全合格。たいせんモードの既存テストも回帰なし

### 追記: バグ修正(ゲスト操作の赤がAIに勝手に蹴られる)

- 症状: ゲストは赤選手を移動できるが、パス/シュートをAIが勝手にやってしまう
- 原因: 8on8は赤全員にAIを走らせてから速度だけ上書きする方式のため、移動は上書きされる一方、holdTimer経過による自動パス/シュート(versusShoot/s7Pass)だけがAIに残っていた(3on3はゲスト選手をAIループから除外済みのため無問題)
- 修正: s7UpdateRedAttackで、保持者がゲスト操作対象(controlledRed)のときは自動キック判断をスキップ。ゲストのタップだけで蹴る。3ケースをユニットテストで確認
- コミット: 6b24f69

### 追記: オンライン Phase 4(切断処理・カクつき補間・ポーズ対応)

- 切断処理: 試合中に接続が切れても画面が固まらないよう onlineHandleDisconnect を追加(close/error/受信5秒タイムアウトで発火)。ゲストはタイトルへ戻して通知、ホストは相手が抜けてもAIとのたいせんとして続行(controlledRedが自動でAIに戻る)。たいせんモードにも同じ検知を追加
- カクつき補間: ゲストは受信位置を目標(net)として持ち、毎フレームそこへ寄せる補間(k=dt*12)で20Hz受信のカクつきをならす。試合開始/再戦の瞬間だけ即スナップ
- ポーズ対応: ホストがポーズ中もブロードキャストを継続(位置は静止)してゲストの誤切断判定を防止。pausedフラグでゲストに「あいてが ちゅうだんちゅう…」を表示
- 検証: 切断・補間のユニットテストを追加し全合格。既存テストも回帰なし
- コミット: aacb9d9

### 追記: しあいモード ゴールキーパー強化(4点)

- GKへのパス: 味方タップ/ミニマップで青GKにもバックパスできるように(候補にs7BlueGK追加、点滅リングもGKに表示)
- GKがボールを持ったら他の選手と同じように自由に操作可能に(s7GiveBallで青GKもcontrolledPlayerにする)。人が操作中はAIの自動ゴールキックを止め、タップで味方へキック

- 検証: パンチ・GK操作権切替をユニットテスト(test_s7_gk)で確認。既存テストも回帰なし

### 追記: オンライン接続が繋がらない問題を修正(重要)

- 症状: 「オンラインであそぶ」で繋がらなくなった
- 調査: PeerJS公開シグナリングサーバー(0.peerjs.com)とライブラリ(unpkg)は生存(HTTP 200・ID発行応答あり)。一方、PeerJS 1.5.4の**デフォルトTURNサーバー(eu-0/us-0.turn.peerjs.com)がDNS解決不可でダウン**していた。STUNは生きているので緩い回線では繋がるが、中継(TURN)が要る厳しいNAT(モバイル/学校)ではP2Pが確立できず接続不可になっていた
- 修正: new Peer に config.iceServers を明示。到達確認できたSTUN(Google/metered)+無料TURN(Open Relay: openrelay.metered.ca 80/443/443-tcp)へ差し替え。443/TCPのTURNでUDP遮断環境も越えやすい。ゲストconnectをreliable:trueに
- あわせて接続確立前の一時エラーで全体が壊れないようガード(onlineConnClosed: ホストは待機継続、ゲストはエラー表示で再試行)。接続待ち画面にヒント文言追加。v33→v34
- 補足: openrelayは無料の共有TURNなので、混雑や将来の停止に備えるなら metered.live 等で専用キー(無料枠あり)を取得して差し替えるのが確実
- コミット: 406ebec
