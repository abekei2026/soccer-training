# 「オンラインたいせんモード」追加開発 設計書

既存の「たいせんモード」(1vs1・攻守切り替え制)を、2台の端末からインターネット経由で対戦できるようにする。

---

## 1. 技術方式

- **通信**: PeerJS (WebRTC DataChannel) を CDN 経由で読み込む。シグナリングは peerjs.com の無料パブリックサーバーを使う(アカウント作成・サーバー構築なし)
  - 唯一の外部依存。index.html は単一ファイルのままだが、`<script src="https://unpkg.com/peerjs...">` を1行追加する
  - 弱点: 一部の厳しいネットワーク環境(学校のWi-Fiなど)ではP2P接続が確立できないことがある。その場合は「せつぞくできませんでした」と表示し、後日TURNサーバー追加を検討する
- **モデル**: ホスト権威型(Host-authoritative)
  - **ホスト**(部屋を作った側)が試合のシミュレーションを100%実行する(既存のたいせんモードのロジックをそのまま流用。もう一方のプレイヤーの入力を、AIの代わりに使うだけ)
  - **ゲスト**は自分のスティック入力・タップだけをホストに送り、ホストから届く状態(ボール・選手の位置、スコア、タイマー)を受け取って描画するだけ(自分では判定・物理演算をしない)
  - これにより既存コードの改変を最小限にできる(ホスト側は「相手をAIではなく通信入力で動かす」だけの差分)

## 2. 通信プロトコル(DataChannel経由のJSON)

- ゲスト→ホスト(入力、毎フレーム送信): `{ t:'in', dx, dy, tap:{x,y}|null }`
- ホスト→ゲスト(状態、20Hz程度で送信): `{ t:'st', ball:{x,y,vx,vy}, me:{x,y,fx,fy}, you:{x,y,fx,fy}, score:{blue,red}, timer, phase }`
  - "me/you" はゲスト視点での自分/相手(送信時にホスト・ゲストの視点を入れ替える)

## 3. 部屋のつくりかた(小学生でも分かる導線)

- たいせんモードの設定画面に「👥 オンラインで あそぶ」ボタンを追加
- 選択肢は2つ:
  - 「へやを つくる」→ 4桁の数字コードを自動生成して大きく表示(例: 3821)。「あいてが はいってくるのを まってるよ…」と表示
  - 「へやに はいる」→ 4桁コードを入力する画面(キャンバス上にテンキーボタンを描画。キーボード操作は使わない=スマホでも迷わない)
- コードは PeerJS の peer ID として使う(衝突を避けるため `st-3821` のような prefix を付ける)

## 4. 実装フェーズ(各フェーズ完了ごとに動作確認・止める)

- **Phase 1**: PeerJS読み込み、部屋作成/入室UI、コード入力UI、実際に2台(またはPCの2タブ)でつながって「せつぞく できました!」が出るところまで(まだ試合はしない)
- **Phase 2**: 接続確立後、ホストが通常のたいせんモードを開始。ゲストは状態(ボール・選手位置・スコア)を受信して描画するだけ(まだゲストは操作できない=観戦できる状態)
- **Phase 3**: ゲストのスティック入力・タップをホストに送信し、ホスト側でその入力を「もう一方のプレイヤー」として反映。双方向に操作できる対戦を完成させる
- **Phase 4**: 切断時の処理(「せつぞくが きれました」→タイトルへ)、コード入力ミスのエラー表示、体感の調整(状態受信の補間などでカクつきを減らす)

## 5. テストについて

- これまでのNode.jsスモークテスト(ブラウザAPIをスタブしてゲームコードを直接evalする方式)は、PeerJS/RTCPeerConnectionが存在しないNode環境では動かないため、`Peer`/`DataConnection`をモックしたスタブを新規に用意してテストする
- 実機確認(実際に2台のスマホ/PCでの接続)は最終的にPhase 3完了時に必ず行う

## 6. 対象範囲(v1)

- 対応するのは「たいせんモード」のみ。しあいモード(7人制)のオンライン対応は、まずたいせんモードで通信の仕組みを確立してから改めて検討する
